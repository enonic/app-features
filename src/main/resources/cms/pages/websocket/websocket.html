WebSocket Session Demo

WebSockets opened within an HTTP session are closed with code 1008 when that session ends — both on logout and on idle-timeout expiry. The behavior is controlled per socket by the terminateOnSessionExit, sessionAccess and sessionAccessThrottleMs parameters of the controller's webSocket response.

The three session sockets connect through an API that only allows role:system.authenticated — while logged out their connections are rejected. Only the public socket connects. Log in, then log out and watch them react.

default

webSocket: {
    data: {...}
}

No session parameters — defaults apply: a socket opened within an HTTP session closes with 1008 when that session ends.

          Note: sockets with no traffic for websocket.idleTimeout (default 5 min, com.enonic.xp.web.jetty.cfg) are closed by the server with code 1001 — that is the connection-level idle timeout, independent of the HTTP session. Turn the pings on to keep connections busy; only the keepalive socket's pings refresh the session.

          How to demo idle-timeout expiry

          Set a short session timeout in com.enonic.xp.web.jetty.cfg: session.timeout = 1 (minutes), restart XP and log in here. With the ping toggle ON, the keepalive socket's inbound messages keep the session alive and all sockets stay open. Switch the pings OFF and stay idle: the session expires after a minute and the scavenger (which sweeps at 1/10 of the session timeout, at least every 10 s) invalidates it moments later — the default and keepalive sockets close with 1008 while the survivor and public sockets stay open.
